## 8. Concurrency limit recommendations (revised)

The `f/check_buddy/daily_payment_sync` script revealed the right pattern: **shared concurrency keys**. Instead of per-script limits, group scripts by what external API they hit and assign one key.

Recommended keys: | Key | Scripts using it | concurrent_limit | time_window_s | Rationale | |---|---|---|---|---| | `qbo_api` | Every script that calls QBO (refresh_*, pull_qbo_*, process_work_order, pre_process_invoice, reconcile_payments, cdc_reconciler, check_buddy/*) | **5** | 10 | QBO realm-level rate limit is ~500/min; 5 concurrent at most. Already used by `f/check_buddy/daily_payment_sync` (set to 1, can stay 1 if conservative) | | `qbo_writer` | Scripts that WRITE to QBO (process_work_order, push_invoice_edits, apply_credit_manual, autopay flow) | **1** | 5 | Serialize all QBO writes to avoid double-charges / race conditions | | `gmail_api` | send_email, send_pending_system_alerts, comms/* | **3** | 10 | Gmail API quota is generous; 3 parallel sends is fine | | `ion_chromium` | All ION flows (visits, work_orders, consumables_usage) | **1** | 7200 | One browser session at a time, full schedule window | | `openai_api` | pre_process_invoice (memo generation) | **5** | 10 | OpenAI rate limits per-key | | `intuit_payments` | process_work_order's charge/echeck calls | **1** | 5 | Money movement; one at a time | These are assigned to scripts via the script's metadata (set via Windmill UI or the API), NOT via inline code.
