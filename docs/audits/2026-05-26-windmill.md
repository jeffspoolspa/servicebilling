## Confirmed runaway: `f/service_billing/pull_customer_payment_methods`

**Captured pattern (2026-05-26T16:30:10)**: the `label-local` Gmail-label trigger
spawned **30+ concurrent jobs in a 250-millisecond burst**. All jobs sat in
queue ~68s (workers saturated), then ran in waves of 2.5s and 63s each. Script
is now deleted (returns 404 from getScriptByPath) ‚Äî you cleaned it up.

**Loop mechanism**: Gmail label-watcher fired N times for N labeled messages,
script had **no `concurrent_limit`** set, so all N runs spawned simultaneously.
Each ~60s QBO call piled up backlog faster than workers could drain. Same
structural risk exists on every `label-*` triggered script in `f/check_buddy/`.

**Fix template for ALL label-triggered scripts** (apply to `f/check_buddy/invoice_match`,
`create_qbo_payment`, `read_qbo_payment`, `search_qbo_payments`,
`get_customer_open_invoices`):

1. Add `concurrent_limit: 1` + `concurrency_time_window_s: 60` to script settings
2. Have the script remove/mark-read the Gmail label on success (or the watcher will keep re-firing on the same message)
3. Add a `WHERE message_id NOT IN (already_processed)` idempotency table guard at the top of `main()`
