Visit Sync — Architecture & Data Flow

How ION pool-service data becomes clean, route-attributed, normalized visits that tools can build on. Status: draft for review · 2026-06-10.

1 · Capture flow + reconcilers

Two clocks. A capture flow (daily ~5 am ET) pulls ION and lands raw facts, then does the cheap DB-only resolution. Independent reconcilers own the table invariants ("every visit has a task", "nothing unmapped") and run on their own cadence — so heavy / fragile per-task ION lookups never gate the daily pull. Everything keyed & idempotent; re-runs never duplicate.

recurring_tasks = standalone reusable flow (also fired after a reroute). ingest only CAPTURES raw (+ grid-tech, which only exists during the scrape). normalize_visits resolves every DB-only FK (tech, consumable, reading, checklist, task_id-from-schedules), is re-runnable after any alias change, and runs off the chromium worker. The task reconciler (ION create for unknown EventIDs) and drift watcher are separate jobs that read the owned tables on their own clock — not steps of the scrape.

4 · The normalization pattern (3 parallel resolvers + a watcher)

Every visit child follows the same shape: ION emits a raw name/id, an alias table maps it to a canonical definition, and a resolved FK is stored next to the raw value. A watcher flags anything raw that has no alias yet, so the approved lists never silently drift.

Drift watcher — what it flags (daily)

  • New unmapped: any ion_item_id / reading name / checklist task_name appearing in the last window with no alias row (or kind='unknown') → surface for a one-line mapping decision (append-only).
  • Reading quality: readings outside reading_definitions.min/max_acceptable → flag (water-chemistry exceptions, and a hook for "should have used corrective_consumables").
  • Sign-out vs usage drift: the chem reconciliation gap (already built) is the inventory-side watcher — extend to a daily/weekly alert per branch.

Output: a small ion.alias_drift queue (or a daily digest) + an alert. Mapping stays a human decision; the watcher just guarantees nothing falls through.

5 · Routes

A "route" today = (office, tech, day_of_week) derived from task_schedules; v_routes_summary already rolls it up (stop counts, frequency mix, value). ION's own route_name/zone ride on ion.recurring_tasks.

A visit attaches to its route through visits.task_schedule_id → the schedule's tech + day + office. So "every visit tied to a route" depends on task_schedule_id being reliably set — that's what step 3 (resolve/relink) does, and it should be folded into the daily run so new visits are route-attributed same-day.

Decide: keep routes derived (view over task_schedules, simplest, already audited) — or promote to a first-class maintenance.routes table (stable route id/name, capacity, color) if tools need to reference a route as an entity. Recommendation: stay derived until a tool needs a durable route id.

6 · History & freshness

  • Visits — immutable grain (one row per LogID). Re-pull corrects in place; the row is the historical record. No SCD needed; add a visits_audit only if we need to track post-hoc edits.
  • Tasks & schedules (routes) — change over time (reroutes, price changes) and are already audited via tasks_audit / task_schedules_audit (before/after/diff jsonb). That's our route-history.
  • Canonical lists & aliases — append-only; aliases are never deleted (techs change route prefixes, ION renames items). History is the append log.
  • Freshness contract — "current through yesterday by ~5:30 am ET, self-healing on a 14-day window." Tools can assume visits ≤ 1 day stale and fully resolved (tech, chemical, task, route).

7 · Open decisions / build queue

A. Fold reading + checklist resolvers into ingest (mirror consumables): backfill existing reading_id/checklist_id, then resolve inline going forward. Requires curating the approved reading_definitions / task_definitions lists first.
B. Build the drift watcher — daily check + flag queue for unmapped items/readings/checklist + out-of-range readings. Decide the alert channel (email? a dashboard table?).
C. Route attribution — fold task_schedule_id relink into the daily run so visits are route-tagged same-day; confirm derived-route vs a routes table.
D. Cadence sign-off — daily 5 am visit sync; weekly route/schedule + non-active task sync; daily drift watcher. Confirm times.
E. Legacy cleanup — retire wide chem_readings in favor of tall visit_readings; confirm/drop stray ion.visit_tasks.
F. Item list completeness — the 25-item chemical list covers ~99% of volume; decide whether to extend canonical chemicals to the tail (DICHLOR, LP-MAX, shocks, stabilizers) so the drift watcher's "unmapped" set shrinks to truly-new items.
