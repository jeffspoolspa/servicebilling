Task — the service contract

A task is one maintenance customer's standing service contract at one place. An active task means an active maintenance customer. It is the engine of the whole maintenance process: signup creates a task, the task's schedule generates visits, and every month the task is active it owes exactly one invoice.

Why this doc exists Code visibility + getting what's in our heads onto paper. Each section maps a concept to the exact place it lives in code or the database, so relationships are forced to be concrete and debugging starts from "where do I look," not "what even is this." Green pills mark facts verified live against the database; amber pills mark drift / unsettled.

What it is & what it's telling us

Reading one task answers: is this place under contract right now, since when, and on what commercial terms — who pays (its service location → customer), how often we come, who is supposed to come, what we charge, what chemicals are expected to cost, and what's bundled into the price.

It is the "should" against which visits (the "did") and invoices (the "billed") are checked. Its single most important job:

  • every completed visit can be attributed to a paying contract — no task means the visit is unbillable;
  • the monthly billing run can ask "did every active contract get invoiced this month?"

The shape — two tables, one concept

A task is a thin header plus one or more schedule slots. The commercial and operational specifics live on the slots, not the header — this is the key structural fact to internalize.

Why price/cadence/tech live on the slot, not the header

Because one contract can recur in several distinct ways at once:

  • A weekly customer serviced Mon + Fri is one task, two slots.
  • A community with an $85 pool contract and a $50 chemical-testing contract is, in our model, one task with two slots at different rates — because of the one-open-task rule below.
Consequence "What does this customer cost?" is never a single number on the task — it's the sum across the task's active slots. Billing reads the slots.
One open task per location — rule the DB enforces

A partial unique index tasks_one_open_per_loc allows only one task with status active/paused per service location. So when ION shows a second contract at a place that already has an open task, the sync attaches it as another slot rather than creating a second task — the "merged" shape (15 legacy tasks bundle several ION tasks this way).

Lifecycle — what flips the status

EventWhat happens
New ION recurring tasktask created active (or a slot attached to the location's open task), with a minimal slot
Still present in IONkept active; financial terms refreshed from the report
Disappears from ION's active reportcancellation — slots go active=false, then task goes closed once no active slot remains
The rule the code encodes but never says out loud On cancellation, ends_on is set to the date of the last actual visit (max(visits.visit_date)), not the sync date — a cancellation ends when service last happened. And we mark closed, never delete (closing also frees the location to take a new contract under the one-open rule).

How it maps to ION — the cleaning pattern

ION is the leader for task existence and financial terms; our model is the clean copy. The translation is an anti-corruption layer: parser → normalize → upsert. Three things to understand:

1. Identity is ionTaskId

ION's "RecurringtasksActive" report returns exactly one row per ION task; ionTaskId is the stable key, stored on task_schedules.ion_task_id, verified 1:1 with our tasks. Every sync update matches on it.

2. The report is a thin financial subset

It has no day-of-week and no tech, and can't see the bi-weekly A/B split. Those (the per-day slots, A/B alternation) came from a richer one-time 2026-04-26 import and the UI — so the ongoing sync deliberately never overwrites day_of_week, tech_employee_id, sequence, or an already-set frequency. ION's full per-task detail (Service Type, Service Profile, Assigned-To grid, Invoice Type, Recurring Note) lives on the ION task screen and is captured separately.

Debugging guide — symptom → where to look

Orphan visit (unbillable) Symptom: a completed visit isn't on any customer's invoice; a customer seems under-billed.
Look: SELECT … FROM maintenance.visits WHERE task_id IS NULL (~2,965 of 19,627 live). Cause: no task at that location, or rate/date inference couldn't pick one. Fix path: f/ION/_lib/taskless_visits.py → f/ION/api/resolve_visit_tasks_via_log.ts (reads the log's EventID) → f/ION/_lib/link_visits_by_event.py.
Visit attributed to the wrong slot Symptom: multi-contract location (e.g. WINDING RIVER: two $85 + two $50 tasks); a visit lands on the wrong rate.
Look: f/ION/_lib/upsert.py:_choose_task and the ambiguous_sl set — same rate + same type can't be told apart from the bulk report. Fix: the EventID correction pass overrides inference with ION's recorded ionTaskId.
A whole customer is missing a task Symptom: a real maintenance customer has visits but no task → all their visits orphan.
Look: f/ION/_lib/upsert_tasks.py unresolved_examples in the sync stats. Cause: the new ION task's address/name didn't match a service location (QBO spelling drift, typo, co-located customers). It is reported, never guessed.
Migrations don't match the live database drift Symptom: reading supabase/migrations/ describes a different shape than reality.
Known drift: live has unique index visits_uniq_log_natural (5-tuple) and columns ion_log_id, ion_calendar_id, ion_addlog_url that exist in no committed migration; the newest repo migration creates a different (3-tuple) index. Trust the live DB (pg_indexes, information_schema.columns), not the migration files, until reconciled.
