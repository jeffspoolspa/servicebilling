ION API — the boundary we pull from

ION Pool Care is an external ColdFusion app with no real API. We wrap it in a thin, typed boundary so the rest of the system calls a function — passes arguments, gets typed rows back — and never touches ION's internals.

What this doc is for (and what it deliberately isn't) This documents the contract: which function we call, what we pass, what we get back, and where each piece lives — so when a pull breaks we know where to look. It does not explain how ION works internally (its sessions, ColdFusion, scraping quirks) — that's ION's concern and the ion-automation skill's. Treat everything past the boundary as a black box.

What this is

Three thin layers of our own code stand between a caller and ION. A caller only ever touches the top one.

  • Endpoints (f/ION/api/*) — one function per thing you can pull. This is the contract.
  • Reports client (f/ION/_lib/reports.ts) — priming, fetching, and HTML→typed normalization.
  • Session (f/ION/_lib/session.ts, session_cache.ts) — login + cookies, cached and reused.

Orchestrator (not an endpoint) f/ION/ingest_day_logs composes the two over a date window, resolves EventID → task, prices at the contracted rate, and does a scoped transactional replace into maintenance.visits + consumables_usage. dry_run default.
Why two calls, not one The day grid (customerLogDetails) is cheap and lists every log, but only the per-log addLog form carries the authoritative EventID (task) + TaskInvoiceID (invoice) + times. That second hop is what makes attribution exact (100% match) instead of inferred.

Inside the box — only when debugging

A caller never needs this. A debugger does: the puller is three sub-steps, and a failure lives in exactly one.

1. Session — session_cache.getOrRefreshSession(ion)

Returns a valid IonSession { cookies, cfClientId, ionOrigin, capturedAt, expiresAt }. Reuses the cached session (Windmill variable f/ION/session_cache) if fresh — no browser — and only logs in via chromium (session.loginToIon, two-stage Fluidra→ION redirect) when stale. Inactivity TTL ~15 min; every successful fetch slides it forward.

In code f/ION/_lib/session_cache.ts:getOrRefreshSession · f/ION/_lib/session.ts:loginToIon (throws if Stage-2 redirect misses ionpoolcare.com)
2. Prime + fetch — reports.ensureReportsPrimed → fetchRecurringTasksHtml

ION report .cfm files read server-side session state the Reports UI sets up; a cold call 500s. ensureReportsPrimed replays the UI's request chain once per session, after which any report fetch is plain HTTP. The recurring-tasks report is GET /reports/_xls/RecurringtasksActive.cfm?techid=0&OfficeID=0&serviceType=0.

In code f/ION/_lib/reports.ts (ensureReportsPrimed, fetchReportHtml, fetchRecurringTasksHtml) · transport via session.ts:ionFetchText
3. Normalize — reports.normalizeRecurringTasks(html)

Finds the data table by the header marker "Cust ID" (parseReportTable), then maps each row's cells by position into a RecurringTask using the fixed RT_COLS order.

The one real leak brittle Mapping is positional and only checks that a "Cust ID" row exists — there's no per-column header assertion. If ION reorders or inserts a column, rows mismap silently (e.g. price lands in the wrong field) with no error thrown. This is the most likely "data looks wrong but nothing crashed" cause. Candidate hardening: assert header names against RT_COLS.

Adding a new ION pull (the pattern)

  1. Reports client (f/ION/_lib/reports.ts): add fetch<Thing>Html(session, filters) (the .cfm path + default params), normalize<Thing>(html) (a typed interface + positional column list), and get<Thing>(session, filters) to compose them.
  2. Endpoint: add f/ION/api/list_<thing>.ts whose main(filters) does getOrRefreshSession → get<Thing>. That's the new black-box function.
  3. Mapper: add f/ION/_lib/upsert_<thing>.py turning the typed rows into our tables; default dry_run=true.
  4. Flow: wire step a (endpoint) → step b (mapper), passing results.a as the rows.

Priming for a different report family would register its own prime chain in ensureReportsPrimed; the scheduled-tasks/service-events family is already covered.
