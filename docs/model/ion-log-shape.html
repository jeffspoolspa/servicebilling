ION service-log — the raw shape

Every field the per-day log pull returns from ION, and which of our tables it feeds. This is the input to the normalize-after step: the API pulls these faithfully; a separate function classifies each into visit / chem_readings / visit_tasks / consumables_usage using the alias tables.

Method & coverage Compiled by probing 30 logs across 10 distinct ION service profiles (CHEMICAL TESTING, FLAT RATE, FOUNTAIN, GREEN POOL, ONE TIME, POOL MAINTENANCE tiers; pools, spas, fountains, kiddie, etc.). Frequencies below are "appeared in N of 30 logs." Rare profiles may add fields not seen here — which is exactly why normalization auto-appends unseen labels rather than hard-coding a list.

The field ids are NOT stable — key on the LABEL The same reading/task gets a different field<digits> id under different service profiles (e.g. "Cleaned Filter" = field14 in one profile, a different id in another). Always read the field's row label, never the numeric id. Verified across all 10 profiles in the sample.

Decided — ingest tag map & rules

Finalized 2026-06-09. Ingest classifies each field by its HTML control (no canonical lookup needed at ingest); every detail row stores the raw name + value + a nullable canonical FK that is backfilled afterward.

Two rules that make this work 1. Readings are tall. One row per reading per visit — visit_readings(visit_id, name, value, reading_id→reading_definitions NULL) — same shape as checklist and consumables; never a widening column set. 2. Name now, FK later. Ingest writes the raw ION label and leaves the canonical FK NULL; after the full load a matching pass fills the FK where the name maps to a definition. Unmatched rows stay with a NULL FK — captured, just outside the normalized analysis set.

Every visit must link to a task

The log's EventID (= the ION task) is stored as ion_task_id on every visit, unconditionally. task_id (our FK) is resolved where the task exists and left NULL otherwise — for now. After a load, EventIDs not yet in maintenance.tasks are looked up in ION and inserted (task + schedule), then task_id is backfilled — so every visit ends up linked. Once the daily puller is in place, "task_id present" becomes a required check.

The tag buckets

General (→ visit row): LogID, calendarID, EventID, TaskInvoiceID, ConsumableInvoiceID, CustomerID, LocID, ScheduledDate, LogDate, timeinvalue, timeoutvalue, InvoiceType, ServiceProfile, OriginalFailureID, submittedBy (tech), comment (notes), failureid (non-service reason).

Ignore: WOadd, SendLogEmail, CanSendEmail, CanReceiveEmail, NonServiceEmailByTask, TaskLocked, ConsumablesLocked, InvoiceConsumables, InvoiceAllVisits, Source, OfficeID (office comes from the customer, not the log), OriginalLogDate, wotempid, LogSubmit. Read only as profile metadata when building canon: fieldlist, ReqFieldList, ReqLabelList, itemlist; recurringNote is task-level, not visit.

Readings and checklist are the field<n> labels in the per-category tables below. By the not-yes/no→readings rule, "Customer Tabs", the Adjustment fields, tank levels, Resident's Chemicals, % Salt Output, ORP, and "Emergency Phone Working?" are all readings — even though some were historically filed as tasks.

Consumables → consumables_usage

Hidden item<digits> inputs whose value (quantity) > 0. The number in the name is ION's item id. In the sample, 7/30 logs had consumables (12 item lines total).

Resolved — the name comes off the log row Each used item renders as <td><strong>SALT 40LB</strong><br>desc</td> next to its hidden item<id> qty input, so get_log_detail reads the name straight off the log — authoritative and complete, no catalog dependency, and it covers ids the mobile qbitems.cfm catalog snapshot misses. Ingest stores ion_item_id + item_name + quantity. The only post-load step is matching item_name → public.items to set the canonical item_id FK (no ion_item_id column on public.items; match by name).

Backfilling the canonical link (the after-step)

Ingest tags by control (see rules) and stores the raw name with a NULL canonical FK. After the full load, a matching pass fills the FK:

Row typeMatch raw name againstSets FK
readingion.reading_aliases → reading_definitions (or slug)visit_readings.reading_id
checklistion.task_aliases → task_definitionsvisit_tasks.checklist_id
consumableION item id / name → public.itemsconsumables_usage.item_id
no match—FK stays NULL; append the raw name to the alias table as pending → assign a canonical later

A label ION renders inconsistently (full label vs short code, select vs text) never blocks ingest — it just may not get a canonical FK until an alias is added. Analysis runs off the FK; everything is captured regardless.

Alias coverage & what's NEW

The alias tables today hold bulk-report short codes (Alk→total_alkalinity, FC→free_chlorine); addLog emits full labels. Most full labels slug straight to the canonical name, but these need explicit alias rows added before/with the backfill:

addLog labelCanonicalWhy slug fails
% SALT OUTPUTsalt_output_pctslug = salt_output
Current Filter PSIfilter_psislug = current_filter_psi
OXIDATION-REDUCTION POTENTIALorpslug = oxidation_reduction_potential
Emergency Phone Working?— (decide: new reading or inspection task)not in any catalog

Checklist labels and the ~15 other readings above all resolve by slug or existing alias — so the "build the canonical tables from observed data" step is small: add the handful of full-label aliases the normalizer reports as unmatched, then re-run.

Decisions (resolved 2026-06-09) & remaining

  1. Readings store: TALL. visit_readings(visit_id, name, value, reading_id→reading_definitions NULL) — one row per reading, name now / FK later. Supersedes the 7-column chem_readings for the log path.
  2. Classification: by control. Yes/no radios → checklist; everything else (select/text) → readings; item<n> → consumables. "Customer Tabs", adjustments, tank levels, % Salt Output, ORP, "Emergency Phone Working?" are all readings.
  3. Office: not captured from the log — derived from the customer. OfficeID ignored.
  4. Every visit links to a task. ion_task_id (EventID) on every visit; unknown EventIDs looked up in ION and inserted into maintenance.tasks/task_schedules after the load, then task_id backfilled; required once the daily puller is live.
  5. Remaining: consumables item resolution (no ion_item_id on public.items; 590 unresolved) needs an item lookup/alias decision; add the 3 new reading aliases (% SALT OUTPUT, Current Filter PSI, OXIDATION-REDUCTION POTENTIAL).
